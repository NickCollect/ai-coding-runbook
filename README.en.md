# AI Coding Runbook · Master Guide

**English** | [中文](./README.md)

> **Written for any LLM and your future self.** Next time anyone — human or AI — opens this project without context, **read this first.** All the rules, lessons learned, and maintenance procedures are here. `CLAUDE.md` / `AGENTS.md` / `system_instructions.md` are subsets / references of this document.

## 1. What This Project Does

**One-liner**: automatically pull official docs from Anthropic, OpenAI, Google, and Cursor, then have an LLM enrich them into a **queryable + cheatsheet-equipped** multi-vendor knowledge base.

**Sources covered**:
- **Anthropic / Claude**: docs.claude.com, anthropic.com blog/research, anthropics + modelcontextprotocol GitHub repos
- **OpenAI / Codex**: docs.openai.com, openai/codex GitHub repo, openai/model_spec
- **Google Gemini**: ai.google.dev Gemini API docs
- **Cursor IDE**: cursor.com/docs (full llms.txt)

**Core use cases**:
1. Ask "Claude Code vs Cursor vs Codex / how to configure hooks / how to use extended thinking" → LLM reads enriched entity files directly (no live googling)
2. Find out what each vendor changed this week → check `03_Output/Changelog/<latest>.md` (auto-generated by GHA)
3. Need a quick reference → grab from `03_Output/Cheatsheets/`
4. New raw arrives (GHA cron runs every Monday) → on session start you'll see "📋 Pending: N files" → user authorizes, LLM processes

**Target audience**: any LLM (Claude Code / Codex / Cursor / others) or human maintainer who picks up this knowledge base.

## 3. Quickstart

> Three most common usage patterns, ordered by setup cost (low → high).

### Pattern 1: Browse the docs (zero config)

```bash
git clone https://github.com/NickCollect/ai-coding-runbook
cd ai-coding-runbook
```

Then open with Obsidian / VSCode / any markdown editor:

- **Cheatsheets** → `03_Output/Cheatsheets/*.md` (hook recipes, API quick refs, model pricing, etc.)
- **Side-by-side comparisons** → `02_Wiki/Comparison/*.md` (Claude / Cursor / Codex decision tables, etc.)
- **Specific feature** → `02_Wiki/Entities/<feature>.md`
- **What changed this week** → `03_Output/Changelog/<latest>.md`

`.obsidianignore` already excludes large directories so Obsidian won't choke.

### Pattern 2: Use as long-term context for an AI agent (recommended, zero config)

After cloning, open the folder in **Claude Code / Cursor / Codex CLI**:

- Session startup auto-loads `CLAUDE.md` / `AGENTS.md`, so the agent knows the project structure and working rules
- Just ask questions; the agent reads `02_Wiki/` to answer. Examples:
  - "How do I configure hooks? Give me a PreToolUse recipe."
  - "Skill vs MCP vs Subagent — which one should I use?"
  - "What did Anthropic / OpenAI change this week?"
  - "Compare Codex CLI and Claude Code."

No additional API key needed — the agent uses your existing subscription / token.

### Pattern 3: Fork to track your own sources (requires a fork)

If you want to:

- Add internal / team doc sources, remove sources you don't need
- Run your own GHA cron (auto-refresh raw every Monday)
- Modify the enrichment workflow

→ Fork this repo to your own GitHub account. The GHA workflow uses the default `GITHUB_TOKEN` — no extra secrets required. Edit the source list in `scripts/sources.yaml`; the next cron run picks it up.

```bash
# Manual local refresh (for debugging)
pip install -r scripts/requirements.txt
python3 scripts/refresh_raw.py --all      # ~10 minutes
```

## 4. Core Mechanisms

### Mechanism 1 · GHA cron auto-pulls raw (matrix parallel)

`.github/workflows/refresh-raw.yml` runs every Monday 01:00 UTC (= 09:00 HKT). **Six sources run in parallel** (GHA matrix), each committing + pushing its own slice (`git pull --rebase` + 5 retries to handle concurrent pushes).

```
matrix sources (parallel):
  - code.claude.com           → 124 docs   (~2 min with concurrency=5)
  - platform.claude.com       → 1275 docs  (~7 min)
  - anthropic.com             → 345 docs   (~2 min, no .md probe)
  - support.claude.com        → 347 docs   (~2 min)
  - github.anthropics         → 12 repos   (~3 min)
  - github.modelcontextprotocol → 6 repos  (~2 min)
```

Within each source: `ThreadPoolExecutor(5)` for concurrent HTTP fetch + auto-retry (429/5xx with backoff). Wall time ≈ max(single source) ≈ **~10 minutes**.

**Aggregator job** (runs once after the matrix completes): scans bot commits in the last 2h → writes `03_Output/Changelog/YYYY-MM-DD.md`.

**`fail-fast: false`** — if one source fails, the others keep going; what was already fetched stays committed.

**Manual trigger**: GitHub repo Actions page → "Run workflow", or locally `gh workflow run refresh-raw`.

**Local refresh** (debugging):
```bash
python3 scripts/refresh_raw.py --list                       # list all source names
python3 scripts/refresh_raw.py --source code.claude.com     # single source
python3 scripts/refresh_raw.py --source github.anthropics
python3 scripts/refresh_raw.py --all                        # all sequentially
python3 scripts/refresh_raw.py --source X --dry-run
```

GitHub repos pulled in get their **`.git/` stripped** (so the parent repo doesn't treat them as submodules). Trade-off: lose the original repo's git history; benefit: files inside are tracked normally by the wiki repo. Full re-clone weekly (small, 3 minutes).

### Mechanism 2 · Enrichment flywheel (cooked dishes stand on their own)

The entity / concept / summary files in `02_Wiki/` are written by an LLM **into those files** after distilling raw — they **already exist as standalone artifacts**.

**Why this matters**: when answering questions, the LLM **doesn't need to re-read raw every time** — reading the enriched entity is enough. This is why queries are fast and accurate.

**Why we don't auto-enrich**: after pulling raw diff, the GHA **does not auto-call an LLM** to write summaries — it only generates a changelog notification. Reasons:
1. LLM enrichment is prone to hallucination and needs self-review
2. What entity / concept to add is a design decision, not a pipeline step
3. The user sees the changelog in their session, decides which diffs are worth enriching and which to skip

### Mechanism 3 · Template-driven living files (future)

Future: periodic prep / report style template generation. `03_Output/templates/` is currently empty; auto-templates will be added once we have enough cheatsheets.

### Mechanism 4 · Structural audit

`scripts/audit.py` checks internal consistency of `02_Wiki/`:

- Does each summary have frontmatter `source:` pointing to a real raw file?
- Does each entity / concept have frontmatter + at least one section?
- Do `[[X]]` wikilinks point to real files?
- Are entity / concept names duplicated?

After running, writes `02_Wiki/_audit_report--YYYYMMDD.md`.

### Mechanism 5 · Canonical-name reconciliation (typos / multiple names for the same thing)

`02_Wiki/_canonical-names.md` is the **single source of truth** for mapping all variant names that appear in raw to the canonical name used inside the vault.

Examples:
- `Sub-agent` / `Subagent` / `Sub agent` → canonical: `Subagent`
- `Tool use` / `Function calling` → canonical: `Tool use`
- `Slash commands` / `Slash Commands` / `slash-commands` → canonical: `Slash commands`

**Mandatory LLM rules**:
1. Before enrichment, `cat 02_Wiki/_canonical-names.md`
2. Before referencing an unfamiliar name, grep the `aliases` field in `02_Wiki/Entities/*.md` frontmatter
3. Don't make up entity names from your head

## 5. Daily Workflow

### What the user does (no LLM needed)

| Scenario | What to do |
|---|---|
| See what Anthropic changed this week | Open `03_Output/Changelog/<latest>.md` |
| Look up a feature quick reference | `03_Output/Cheatsheets/<topic>.md` |
| Pull remote updates (after GHA cron runs) | `cd <repo dir> && git pull` |
| Add a new source | Edit `scripts/sources.yaml`, commit + push, next cron will pick it up |

### What the user asks the LLM to do

| Scenario | What the user says | What the LLM does |
|---|---|---|
| New raw to ingest | **No need to mention it** — session start auto-detects "📋 Pending". Say "ingest" / "ingest 1,3" / "skip" | Phase A→E (see § 5 ingest) |
| Any Claude Code / API question | Just ask | Reads enriched entity to answer; falls back to raw if missing info |
| Write a new cheatsheet | "Write a hooks cheatsheet" / "Make a Skill vs MCP comparison" | Cross-entity synthesis → write to `03_Output/Cheatsheets/` |
| Add a new source | "Add modelcontextprotocol/inspector repo" | Modify sources.yaml + commit + run a refresh test |
| Run audit | "Run an audit" | `python3 scripts/audit.py` |
| Fix a typo | "The Anthropic doc spells X as Y, but the vault should use Y" | Update canonical-names + bulk replace + re-run audit |

### LLM ingest workflow Phase A→E

After user authorization:

| Phase | What | Self-review |
|---|---|---|
| **A · Create summaries** | Write a summary in `02_Wiki/Summaries/` for each pending raw | frontmatter compliant, source field points to real raw, no name collisions |
| **B · Update existing entities / concepts** | Append to `## Sources`; add new facts to `## Key Properties` | **Don't create new files**; all wikilinks resolve; every fact has source backing |
| **C · Handle new names** | List candidates to user, user picks → create stubs | Only create after user authorizes |
| **D · Audit** | `python3 scripts/audit.py` | Surface fix suggestions; don't auto-fix |
| **E · Log** | Append to `02_Wiki/_progress.log` | — |

Self-review at every phase **prevents earlier-stage hallucinations from contaminating later stages**.

## 6. 🚨 Lessons Learned

### #1 · Never modify 01_Raw

Raw is the GHA bot's output. Manually editing raw causes:
- Next cron run reverts the change (tug-of-war)
- Pollutes the git diff signal
- Misleads enrichment (looks like Anthropic changed something)

For "corrections" (like Anthropic doc typos or stale references) → write to `02_Wiki/_canonical-names.md` and use the errata mechanism.

### #2 · When model weights conflict with raw, raw wins

Anthropic ships API changes / new features frequently; everything after Claude's training cutoff lives only in raw. When answering, if raw has it, defer to raw; only fall back to model knowledge for what raw doesn't cover (and explicitly mark it "based on training knowledge, not verified against latest docs").

### #3 · "I remember Claude Code has this feature" is not evidence

Every fact in an entity / concept must be traceable to a raw file. Before writing "Hooks supports X", grep `01_Raw/` for the original. If you can't find it, mark it `(unverified)` or skip it.

### #4 · One concept, multiple names

Anthropic's own docs are inconsistent: `Sub-agent` / `Subagent` / `subagent`, `Tool use` / `Function calling`, `Slash commands` / `slash-commands`. Reconciliation lives in `_canonical-names.md`. Inside the vault, use the canonical name; aliases go in entity frontmatter.

### #5 · Always `ls` before writing a wikilink

Before writing `[[Hooks]]`, run `ls 02_Wiki/Entities/Hooks.md`. Don't write links to non-existent files — they become dead links. Audit catches them, but you should avoid them at write time.

### #6 · Don't trust subagent findings directly

After dispatching a subagent to research a topic, the master phase **must** sample re-verify (≥3 + every MAJOR claim). Subagents hallucinate, especially when researching territory the vault hasn't yet enriched.

### #7 · GHA pushes to main are fine, but be careful breaking sources.yaml

GHA workflow has no PR review; a bad prefix in sources.yaml can cause the next cron to fetch 0 files / wrong files. Validate with `python3 scripts/refresh_raw.py --dry-run --only <kind>` locally before committing.

### #8 · Multi-machine sync

Sync via git, not iCloud. **Don't hardcode absolute paths in scripts** — use `~`, `$HOME`, or project-relative paths (`Path(__file__).resolve().parent.parent`) so it runs on any machine.

## 8. Maintenance

### Long-term routine (you do)
- After GHA runs each Monday, `git pull`
- Read `03_Output/Changelog/<latest>.md` to decide whether to ingest
- Want to ingest → say "ingest" in a Claude Code session

### Long-term routine (GHA does automatically)
- Every Monday 09:00 HKT, pull raw
- Diff detection + commit + push
- Write changelog
- GitHub email alert on failure

### Adding / changing a source
1. Edit `scripts/sources.yaml`
2. Validate locally with `python3 scripts/refresh_raw.py --dry-run --only <kind>`
3. commit + push
4. Wait for next cron, or trigger `gh workflow run refresh-raw`

### Modifying crawler behavior
1. Edit `scripts/refresh_raw.py`
2. Validate locally with `python3 scripts/refresh_raw.py --dry-run`
3. commit + push
4. Next cron uses the new version

## 9. Glossary

- **raw**: files under `01_Raw/`, fetched markdown by the GHA bot or shallow-cloned repos
- **summary**: under `02_Wiki/Summaries/`, 1:1 summary of each raw file
- **entity**: under `02_Wiki/Entities/`, dossier on a specific feature / tool / model (aggregates multiple raw files)
- **concept**: under `02_Wiki/Concepts/`, an abstract concept
- **synthesis**: under `02_Wiki/Synthesis/`, cross-entity essays
- **comparison**: under `02_Wiki/Comparison/`, decision matrices / side-by-side
- **cheatsheet**: under `03_Output/Cheatsheets/`, outward-facing quick references
- **changelog**: under `03_Output/Changelog/`, auto-written records of raw changes
- **canonical name**: the unified internal name (vs the multiple variants that appear in raw)
- **stub**: placeholder entity / concept marked with `<!-- stub: awaiting enrichment -->`
